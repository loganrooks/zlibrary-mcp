# Contributing to Z-Library MCP Server

Thank you for your interest in contributing! This guide covers everything you need to get started.

## Getting Started

### Prerequisites

- **Node.js 22+** ([download](https://nodejs.org/))
- **Python 3.10+** ([download](https://www.python.org/))
- **UV** -- modern Python package manager ([install](https://docs.astral.sh/uv/))
- **Git**

### Clone and Setup

```bash
git clone https://github.com/loganrooks/zlibrary-mcp.git
cd zlibrary-mcp

# Setup Python environment (creates .venv/ with all dependencies)
bash setup-uv.sh

# Install Node.js dependencies
npm install

# Build TypeScript
npm run build
```

### Verify Setup

```bash
# Node.js tests (Jest)
npm test

# Python tests (pytest)
uv run pytest
```

Both test suites should pass with no errors.

## Project Architecture

This is a dual-language project:

- **Node.js/TypeScript** (`src/`): MCP server implementation, tool registration, client communication
- **Python** (`lib/`): Z-Library EAPI client (`python_bridge.py`) and document processing (RAG pipeline in `lib/rag/`)
- **Vendored fork** (`zlibrary/`): Custom Z-Library EAPI client for search and downloads
- **Compiled output** (`dist/`): TypeScript compiled to JavaScript (generated by `npm run build`)

See the [Architecture Diagram](README.md#architecture-overview) in the README for a visual overview.

### Key Files

| File | Purpose |
|------|---------|
| `src/index.ts` | MCP server entry point, tool definitions |
| `src/lib/zlibrary-api.ts` | Bridge between Node.js and Python via PythonShell |
| `src/lib/venv-manager.ts` | Python virtual environment lifecycle management |
| `src/lib/paths.ts` | Path resolution for Python scripts |
| `lib/python_bridge.py` | Core Python logic for Z-Library operations |
| `lib/rag_processing.py` | Document processing orchestrator |
| `lib/rag/` | Decomposed RAG pipeline modules |

## Running Tests

### Node.js (Jest)

```bash
# Full test suite with coverage
npm test

# Specific test file
node --experimental-vm-modules node_modules/jest/bin/jest.js __tests__/zlibrary-api.test.js
```

### Python (pytest)

```bash
# Full test suite
uv run pytest

# Fast subset (skips slow and integration tests, ~4s)
uv run pytest -m "not slow and not integration"

# Specific test file
uv run pytest __tests__/python/test_rag_processing.py
```

### End-to-End (Docker)

```bash
npm run test:e2e
```

Coverage thresholds are enforced -- CI will fail if coverage drops below the project baseline.

## Code Style

Code style is enforced automatically via lint-staged, which runs on every `git commit`:

- **TypeScript** (`src/**/*.ts`): ESLint (with `--fix`), Prettier, then `tsc --noEmit` for type checking
- **Python** (`*.py`): Ruff check (`--fix`) and Ruff format

You do not need to run linters manually -- they run automatically when you commit.

### Blame-Ignore-Revs

The repository has a `.git-blame-ignore-revs` file for the initial Prettier formatting commit. To get clean `git blame` output:

```bash
# One-time global configuration
git config blame.ignoreRevsFile .git-blame-ignore-revs

# Or per-command
git blame --ignore-revs-file .git-blame-ignore-revs src/index.ts
```

## Making Changes

### Branch Naming

Create a feature branch from `master`:

```bash
git checkout -b feature/your-feature-name   # New feature
git checkout -b fix/issue-description        # Bug fix
git checkout -b docs/topic                   # Documentation
```

### Commit Messages

Use [Conventional Commits](https://www.conventionalcommits.org/) format:

```
feat: add new search filter for publisher
fix: handle empty search results gracefully
docs: update API reference with new parameter
refactor: extract retry logic into separate module
test: add integration tests for download flow
chore: update dependencies
```

Keep commits focused -- one logical change per commit.

## Pull Request Process

1. Create a feature branch from `master`
2. Make your changes, following existing code patterns
3. Ensure tests pass locally: `npm test` and `uv run pytest`
4. lint-staged hooks will run automatically on commit (ESLint, Prettier, Ruff, TypeScript type-check)
5. Push your branch and open a PR
6. CI will run tests, linting, and coverage checks
7. Address any review feedback

### PR Description

Include:
- What the change does and why
- Any breaking changes
- How to test the change

## Code Patterns

### Error Handling

Use Error cause chaining for context propagation:

```typescript
throw new Error('Failed to search books', { cause: error });
```

### Python Bridge Calls

All Python operations go through `src/lib/zlibrary-api.ts` via PythonShell:

```typescript
import { ZLibraryAPI } from './lib/zlibrary-api.js';
const api = new ZLibraryAPI();
const results = await api.searchBooks({ query: 'test' });
```

### Path Resolution

Use the path helpers in `src/lib/paths.ts` for Python script paths:

```typescript
import { getPythonScriptPath } from './lib/paths.js';
const scriptPath = getPythonScriptPath('python_bridge.py');
```

For detailed patterns, see the project's internal documentation in `.claude/PATTERNS.md`.
