Create `CONTRIBUTING.md` at the project root with contributor guidance. This should be standalone (a contributor should not need to read .claude/ docs) but can reference them for deep dives.

**Sections:**

1. **Getting Started**
   - Prerequisites: Node.js 22+, Python 3.10+, UV, Git
   - Clone and setup:
     ```bash
     git clone https://github.com/loganrooks/zlibrary-mcp.git
     cd zlibrary-mcp
     bash setup-uv.sh
     npm install
     npm run build
     ```
   - Verify setup: `npm test` (Jest) and `uv run pytest` (Python)

2. **Project Architecture** (brief)
   - Dual-language: Node.js/TypeScript MCP server + Python bridge for Z-Library API and document processing
   - Key directories: `src/` (TypeScript source), `lib/` (Python bridge/RAG), `zlibrary/` (vendored EAPI client), `dist/` (compiled output)
   - Link to architecture diagram in README

3. **Running Tests**
   - `npm test` -- Jest tests with coverage (Node.js)
   - `uv run pytest` -- Python tests with coverage
   - `uv run pytest -m "not slow and not integration"` -- fast subset (~4s)
   - `npm run test:e2e` -- Docker-based E2E tests
   - Coverage thresholds are enforced -- CI fails if coverage drops below baseline

4. **Code Style**
   - ESLint + Prettier enforced via lint-staged (runs on git commit)
   - TypeScript: `src/**/*.ts` files checked by eslint, prettier, and tsc
   - Python: `*.py` files checked by ruff (check + format)
   - Note: `.git-blame-ignore-revs` exists for the initial Prettier formatting commit -- use `git blame --ignore-revs-file .git-blame-ignore-revs` or configure globally with `git config blame.ignoreRevsFile .git-blame-ignore-revs`

5. **Making Changes**
   - Branch naming: `feature/description`, `fix/description`, `docs/description`
   - Conventional commits: `feat:`, `fix:`, `docs:`, `refactor:`, `test:`, `chore:`
   - Keep commits focused -- one logical change per commit

6. **Pull Request Process**
   - Create a feature branch from `master`
   - Ensure `npm test` and `uv run pytest` pass locally
   - lint-staged hooks will run automatically on commit
   - Open a PR with description of changes
   - CI will run tests, linting, and coverage checks

7. **Code Patterns**
   - Error handling: Use Error cause chaining (`throw new Error(msg, { cause: error })`)
   - Python bridge calls: Use PythonShell via `src/lib/zlibrary-api.ts`
   - Path resolution: Use `src/lib/paths.ts` helpers for Python script paths
   - For detailed patterns, see the project's internal documentation

8. **Reporting Issues**
   - Use GitHub Issues
   - Include: Node.js version, Python version, OS, error output, steps to reproduce
