**1. Create `.github/workflows/publish.yml`:**

Per locked decisions: manual trigger (not semantic-release), triggered by version tags, requires NPM_TOKEN secret.

```yaml
name: Publish to npm

on:
  push:
    tags: ['v*']
  workflow_dispatch:

permissions:
  id-token: write
  contents: read

jobs:
  publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          registry-url: 'https://registry.npmjs.org'
      - uses: astral-sh/setup-uv@v4
      - run: npm ci
      - run: uv sync
      - run: npm run build
      - name: Run tests
        run: |
          node --experimental-vm-modules node_modules/jest/bin/jest.js
          uv run pytest -m "not slow and not integration" --benchmark-disable
      - name: Upgrade npm for provenance support
        run: npm install -g npm@latest
      - name: Publish with provenance
        run: npm publish --provenance
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

Key design choices:
- Uses `registry-url: 'https://registry.npmjs.org'` in setup-node to configure .npmrc for auth
- `id-token: write` permission enables npm trusted publishing (OIDC) if configured on npmjs.com. Falls back to NPM_TOKEN secret.
- `npm install -g npm@latest` upgrades npm to 11.x+ for `--provenance` support (Node 22 ships npm 10.x)
- Runs fast tests (not full suite) before publishing -- full suite runs on push to master via ci.yml
- Does NOT use the composite action (publish workflow is standalone, rarely runs, and the composite action includes checkout which publish already handles with setup-node registry-url config)
- `workflow_dispatch` allows manual triggering for dry-run testing of the workflow
